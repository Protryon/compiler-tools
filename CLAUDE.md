# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
cargo build --workspace --all-features      # build everything
cargo test --workspace --all-features       # run all tests
cargo test --package compiler-tools          # runtime-crate unit tests only
cargo test --package compiler-tools-derive --test behavior   # one integration test binary
cargo test --package compiler-tools-derive --test behavior -- spans_track_newlines   # a single test
cargo test --package regex-conformance --test conformance -- --nocapture   # run the regex corpus against both engines; prints failing names + pass/parse/match counts + timings
cargo clippy --all
cargo +nightly-2025-03-01 fmt --all -- --check   # CI pins this exact nightly for rustfmt
```

`rustfmt.toml` uses a wide layout (`max_width = 160`, `struct_lit_width = 0`) — generated/`quote!` code and tests are expected to follow it. Several width options are unstable, so the formatting check requires nightly; the CI `rustfmt` job pins the exact nightly toolchain.

The crates are on `edition = "2024"` with `rust-version = "1.85.0"` (the CI `tests_min_compat` job pins that MSRV — bump both together if you raise it).

## Architecture

A workspace for building fast tokenizers from an enum definition:

- **`compiler-tools`** — the runtime library: `Span`/`Spanned`, the `TokenParse`/`TokenExt` traits, the `TokenizerWrap` helper, and small reusable parse helpers (`util::parse_str`, `MatchResult`).
- **`compiler-tools-regex`** — the compile-time simple-regex engine (the `parse`/`nfa`/`dfa`/`generate` pipeline plus the `matching` runtime interpreter, the `search` regex-crate-shaped runtime API — see *The `RegexSearch` runtime API* below — and `unicode` for build-time Unicode-table lookups via `regex-syntax`). A plain library crate (not a proc-macro) so it can be reused: `compiler-tools-derive` depends on it for code generation, and `regex-conformance` exercises it directly. Generated matchers refer to `::compiler_tools::...` by absolute path. The optional `jit` feature adds a third DFA consumer — `jit.rs` — that lowers the same DFA to native code at *runtime* via Cranelift (see *The Cranelift JIT engine* below).
- **`compiler-tools-derive`** — the `#[token_parse]` proc-macro that generates a tokenizer. Depends on `compiler-tools-regex`, and `dev-depends` back on `compiler-tools` so the integration tests in `tests/` exercise generated code.
- **`regex-conformance`** — a test-only crate (`publish = false`) that runs the upstream `regex` test corpus (`testdata/`) against each form of the engine (the interpreter, the generated-Rust matcher, and — under `--features jit` — the Cranelift JIT); see *Regex conformance* below.

Generated code refers to the runtime crate by absolute path (`::compiler_tools::...`), so the macro output is self-contained at any call site.

### The `#[token_parse]` pipeline (`compiler-tools-derive/src/lib.rs`)

Applied to an enum, the macro emits: the enum re-declared with `#[token(...)]` attributes and string discriminants stripped; a `Display` impl; a `TokenExt` impl (`matches_class`, which compares variant *kind* and ignores payload); and a `XxxTokenizer<'a>` struct implementing `TokenParse`. The tokenizer name is the enum name with `Token` replaced by `Tokenizer`, or `Tokenizer` appended.

Each variant is specified by one of:
- a string discriminant (`Async = "async"`) or `#[token(literal = "...")]` → exact literal
- `#[token(regex = "...")]` → the **custom** compile-time regex engine (see below)
- `#[token(regex_full = "...")]` → the `regex` crate, compiled at runtime into a `OnceLock`
- `#[token(parse_fn = "path::to::fn")]` → a user `fn(&str) -> Option<(&str, &str)>` returning `(matched, remaining)`
- `#[token(illegal)]` → single-char fallback when nothing else matches

A variant's payload determines construction: unit (no payload), `&'a str` (raw matched slice), or any other type `T` (parsed via `passed.parse()`, where a parse failure rejects the match).

### Matching order and literal/regex conflict resolution

The generated `next()` tries matchers in **declaration order** (token index). Literals are compiled into a character trie (`LitTable`, `lit_table.rs`) emitted at the first literal variant's index and matched longest-first; each regex/`parse_fn` emits its own block at its variant's index. First match wins.

The subtle part: a literal that would *also* be matched by a **later-declared** regex (the canonical keyword-vs-identifier case — `let` vs `[a-z][a-zA-Z0-9_]*`) is **removed from the trie** and instead injected as a special-case arm inside that regex's generated parser. So the longer regex match wins (`letx` → identifier), but when the matched text equals the literal exactly it yields the keyword token (`let` → keyword). This conflict detection lives in `impl_token_parse` and uses the engine's `Regex::matches` / the `regex` crate's `Regex::is_match` at macro-expansion time.

### The custom simple-regex engine (`compiler-tools-regex/src/`)

A small regex compiler used by `#[token(regex = ...)]`, producing a branch-only state machine with no per-call allocation (faster than the `regex` crate for token-sized inputs; see `compiler-tools-derive/tests/regex_bench.rs`):

`parse.rs` (pattern → AST of literal/group/**alternation** atoms with `* + ?` repeats, each carrying an `AtomRepeat::lazy` flag for non-greedy `*? +? ?? {n,m}?`; `(...)` groups and `|` are parsed recursively into `Atom::Alternation(Vec<branch>)`) → `nfa.rs` (Thompson construction: each atom/repeat/branch becomes an epsilon-wired fragment, with branching routed through **pure-epsilon split states** whose epsilon-edge *order* encodes thread priority — body-before-exit for greedy, flipped for lazy, earlier-branch-first for alternation) → `dfa.rs` (**priority-preserving** subset construction — each DFA state is a *priority-ordered* epsilon-closure `Vec<u32>` truncated at the accepting state, with consuming edges partitioned into disjoint char classes with **ordered-unioned** targets, so shared-prefix alternation like `a|ab` is deterministic *and* leftmost-first) → `generate.rs` (emits a per-state `match` function returning `MatchResult::{Matched, MatchedEmpty, NoMatch}`). `matching.rs` interprets the same DFA at runtime: `Regex::matches` (a bool prefix check used for conflict detection at macro-expansion time) and `Regex::find_prefix` (returns `(matched, remaining)`, mirroring the generated matcher's semantics — used by the conformance harness). Both `find_prefix` and the generated matcher are **leftmost-first** (like the `regex` crate): they follow the highest-priority surviving thread, remember the last accepting position, and back off to it on a dead end. The accept-truncation in `dfa.rs` is the trick — any consuming edge left in a state outranks that state's accept, so "prefer to consume" is greedy, while a lazy/earlier-branch path has no surviving consuming edge and accepts immediately (e.g. `a|ab` on `"ab"` → `"a"`, `a.*?b` stops at the first `b`). `End` edges only mark a state accepting; the zero-width moves are `EndOfInput` (`$`/`\z`), `StartOfText` (non-multiline `^`/`\A` — holds only when `prev` is `None`), `WordBoundary`, and the `(?m)` line anchors `StartOfLine { crlf }` (`^`) / `EndOfLine { crlf }` (`$`) — `crlf` (set by `(?R)`) widens the line-terminator set those anchors test from `\n`-only to `\r`/`\n`/the atomic `\r\n`. `find_prefix` and the generated matcher take a `prev: Option<char>` (the char before the slice) so a mid-input slice still evaluates `^`/`\b` against the correct preceding context (the conformance search passes the real preceding char; the generated tokenizer passes `None`, so `^`/`\A` is a no-op there). Beyond following the highest-priority thread, both engines also record an accept reachable from the current state purely through zero-width assertion edges that hold *here* (`accepts_via_assertions` in the interpreter; a per-state `if <cond> { last = … }` precomputed by `zero_width_accept_conditions` in the codegen) — so a greedy run backs off to an assertion-gated accept it would otherwise consume past (`.+\b`, `\B(?:fo|foo)\B`). What's still missing is an assertion that must expose a *further consuming* match in parallel with a live greedy thread (`.*\bx`); that needs a multi-thread step, see `REGEX_PARITY.md`.

Supported syntax: literals, `[...]` classes and `a-z` ranges, negation `[^...]`, `.` (any char except `\n`, like the `regex` crate), the quantifiers `* + ?` and counted `{n} {n,} {n,m}` (each in a greedy or lazy `*? +? ?? {n,m}?` form), control-char escapes (`\n \t \r \0 \f \v`), hex/codepoint escapes (`\x41`, `\uXXXX`, `\UXXXXXXXX`, and braced `\x{..}`/`\u{..}`/`\U{..}`), Perl shorthand classes (`\d \D \w \W \s \S` — ASCII by default, full Unicode sets under `(?u)` via `regex-syntax`; a negated shorthand inside a `[...]` class is materialised as its complement), Unicode property classes `\p{..}`/`\P{..}` (and the `\pL` shorthand; resolved to codepoint ranges at build time via `regex-syntax` in `unicode.rs`, usable top-level and inside `[...]`), `^`/`\A` (non-multiline start-of-text, lowered to `Atom::StartOfText` — a zero-width assertion that holds only at the very start of the input, `prev == None`; valid anywhere, not just leading), `$`/`\z` (end-of-input assertion, lowered to `Atom::EndOfInput`/`TransitionEvent::EndOfInput`; valid anywhere, not just trailing), and word boundaries `\b`/`\B` plus the directional half-boundaries `\b{start}`/`\b{end}`/`\b{start-half}`/`\b{end-half}` (`Atom::WordBoundary { kind: WordBoundaryKind, unicode }` — `kind` selects the per-side condition over the word-ness of `prev`/lookahead, `\b{…}` directives parsed in `parse_boundary_directive`; ASCII `[0-9A-Za-z_]` word-ness by default, Unicode `\w` word-ness under `(?u)`; the Unicode set emitted once as a binary-searched range table), and `(?m)` multiline `^`/`$` (`Atom::StartOfLine`/`EndOfLine`: `^` holds at start-of-input or after a line terminator, `$` at end-of-input or before one, and they may appear anywhere/quantified; under `(?R)` CRLF mode the terminator set is `\r`/`\n`/the atomic `\r\n`, so the anchors don't split inside a `\r\n` pair, and `.` excludes `\r` as well as `\n`). The word-boundary/anchor zero-width assertions are evaluated in the generated matcher loop, which holds a lookahead char + `prev` and only advances on a consuming match (so `MatchedEmpty` transitions are truly zero-width). **Alternation `a|b` and grouping `(...)`** are supported, including nesting, a quantifier on a group (`(ab)+`), and non-capturing/named groups (`(?:...)`, `(?P<n>...)`, `(?<n>...)` — all treated identically since this engine never extracts capture spans). **Inline flags / modes** are supported — both the bare `(?flags)` directive (applies to the rest of the enclosing group) and the scoped `(?flags:...)` group, with `-` clearing: `i` (case-insensitive — a cased literal becomes a multi-case group and class members/ranges are folded; ASCII folding by default, full Unicode simple case folding under `(?iu)` via `regex-syntax`, see `unicode::case_fold`), `s` (dot-all — `.` lowers to an inverted empty class), `x` (verbose — drop unescaped whitespace + `#` comments outside `[...]`), `U` (swap-greedy), and `u` (Unicode toggle — sets a `unicode` flag, defaulting *off* so `#[token(regex = ...)]` stays ASCII; when on, `\d \w \s` use their full Unicode sets, `\b`/`\B` use Unicode word-ness, and `(?i)` does Unicode case folding). `m` (multiline — `^`/`$` become line anchors, see above), and `R` (CRLF — widens the `m` anchors' terminator set to `\r`/`\n`/`\r\n` and makes `.` exclude `\r`). See `Flags`/`apply_flag`/`fold_entry` in `parse.rs`. Lookaround is rejected at parse time. Still missing: POSIX classes, octal escapes, custom (non-`\n`/CRLF) line terminators — use `regex_full` for those. `REGEX_PARITY.md` tracks the remaining gaps against the `regex` crate.

### The `RegexSearch` runtime API (`compiler-tools-regex/src/search.rs`)

The engines expose a single anchored primitive — `find_prefix(from, prev) -> Option<(matched, remaining)>`, on both `Regex` (the DFA interpreter) and `JitRegex` (the Cranelift JIT). `search.rs` layers a `regex`-crate-shaped *search* API over that primitive via the `RegexSearch` trait: one required method (`find_prefix`, which each engine implements by forwarding to its inherent method) and default methods for the orchestration the `regex` crate's `Regex` adds — `is_match`/`is_match_at`, `find`/`find_at` (returning a `Match` with `start`/`end`/`range`/`as_str`), `find_iter` (leftmost-first, non-overlapping, with the `regex` crate's empty-match-adjacency rule), `replace`/`replace_all`/`replacen`, and `split`/`splitn`. Both engines share the one implementation, so the API is identical across the interpreter and the JIT (bring the trait into scope to call it). This is distinct from `Regex::matches`, the unanchored *prefix-only* bool check used at macro-expansion time for keyword/identifier conflict detection. **Capture groups are deliberately absent** (`captures`, a `Captures` type, `$1`/`$name` replacement expansion): the engine discards capture-span identity during NFA construction, so exposing them needs engine work — see the *API parity* section of `REGEX_PARITY.md`, which is also where the replacement helpers' literal-string (non-`Replacer`) contract is recorded.

### Regex conformance (`regex-conformance/`)

The `testdata/` directory holds the upstream `regex` crate's TOML test corpus, parsed with the `regex-test` crate. The `regex-conformance` crate runs every test against **both** forms of the engine and prints a per-engine summary (the two should always agree):

- the **runtime interpreter** — `Regex::find_prefix`, walking the DFA directly;
- the **compiled-Rust engine** — `fn(&str) -> Option<(&str, &str)>` matchers that `regex-conformance/build.rs` code-generates via `Regex::generate_parser` (the exact code `#[token(regex = ...)]` expands to), one per test, looked up by full name.
- the **Cranelift JIT engine** (only under `--features jit`) — `Regex::compile_jit().find_prefix`, native code built at runtime (see *The Cranelift JIT engine*). Unlike the first two (a bring-up comparison that never fails), the JIT column **asserts** it matches the interpreter exactly — same pass/skip counts and identical fail-to-parse/fail-to-pass sets — since both walk the same DFA, so any divergence is a lowering bug and fails the test.

All engines are anchored prefix matchers, so the harness (`run_search` in `src/lib.rs`) turns each into a leftmost, non-overlapping search to match the corpus' expectations. The `build.rs` codegen is why the engine had to be a normal crate rather than living inside the proc-macro crate. Each test lands in one of: pass, fail-to-parse (parser rejected it), fail-to-pass (parsed but wrong matches), or skipped (regex set / non-UTF-8 haystack). The test always passes for the first two engines — it's a bring-up harness, not a gate — and prints failing test names plus counts and timings under `--nocapture`. `REGEX_PARITY.md` records the latest tallies.

### The Cranelift JIT engine (`compiler-tools-regex/src/jit.rs`, `--features jit`)

A third consumer of the `Dfa`, alongside the `matching` interpreter and the `generate` Rust-source codegen: it JIT-compiles the DFA to machine code at *runtime* via Cranelift (the runtime-compiled analogue of `generate.rs`, and the runtime sibling of `#[token(regex_full)]`). Off by default — Cranelift is a heavy tree — and pinned to `cranelift-* 0.121.x`, whose `rust-version` is exactly the workspace MSRV (1.85.0); newer Cranelift raises its MSRV in lockstep with the Rust release train, so bumping it means bumping the workspace MSRV. `Regex::compile_jit()` returns a `JitRegex` whose `find_prefix(from, prev)` matches the interpreter's signature/semantics; it owns the `JITModule` (executable memory) for the function pointer's lifetime.

The compiled matcher is `extern "C" fn(ptr, len, prev: u32) -> usize`: it returns the byte offset of the last accept or `usize::MAX` (the same sentinel `generate.rs` uses), and `prev` is the preceding codepoint with `u32::MAX` for `None`; the Rust wrapper rebuilds `Option<(&str, &str)>`. Requires a 64-bit host (the cursor arithmetic assumes it). **Lowering model**: each DFA state is one Cranelift block, and a state's consuming edges branch *directly* to their target state's block — the "current state" is just which block we're in, no runtime state variable or switch. The mutable values are SSA `Variable`s (`counter`/`last`/`prev`/`zero_width`); the lookahead codepoint + width are *not* variables — each state block decodes the char at `counter` inline at its top (`Lower::decode_at`, a UTF-8 length-class diamond converging on a `(cp, width)` block-param `merge`) and threads those SSA values through the block, so there is **no per-character helper call**. It mirrors `generate.rs` arm-for-arm: consuming edges win over zero-width ones; an accepting state records `last = counter` and a non-accepting state records it when an accept is reachable through zero-width assertions holding here (the `.+\b` backoff, via `assertion_accept_paths` + a branchless `select`); the zero-width fallback takes the highest-priority holding assertion as a non-consuming move with the same cycle guard. The consuming dispatch is *not* a linear edge scan: the DFA's `partition` guarantees a state's consuming edges are disjoint sorted codepoint ranges, so they're flattened into one `(lo, hi) -> target` table (`consuming_ranges`) gated by a single `has_char` (end-of-input) check, with no eager full-class membership. The table splits into singletons and wider ranges: a dense singleton set (≥4, the keyword/literal-alternation case) lowers to a Cranelift `Switch` (jump table / tuned bsearch), and the disjoint wider ranges to a binary search (`emit_range_search`, O(log n) compares); a `Switch` miss falls through to the range search. Only Unicode `\w` word-ness is still delegated to an imported Rust helper (`jit_is_word_unicode`), since the table is large and `\b` is comparatively rare. On a long input the JIT runs ~4.8× faster than the interpreter (`jit::tests::jit_timing`); on the conformance corpus it's roughly on par with the `regex` crate (the binary-search dispatch cut its time ~28%) though still a few× the LLVM-compiled `generate.rs` matcher — Cranelift trades runtime quality for compile speed. (Two further levers were tried and dropped/kept on evidence: a `Switch` jump table for dense singleton states — kept, correct, though regex-shaped corpus patterns rarely trigger it; and raising the ISA `opt_level` to `speed` — reverted, since it slowed codegen markedly for no measurable runtime gain on token-sized matchers.) The conformance harness's JIT column asserts full parity with the interpreter across the corpus.

### Span tracking

Every matcher computes a `Span { line_start, col_start, line_stop, col_stop }` and advances the tokenizer's running `line`/`col`. `Span`'s `PartialEq` always returns `true` and `Hash` is a no-op, so `Spanned<T>` tokens compare by value, not by location. Matchers pick a newline-counting path vs a fast column-only path at compile time via `Regex::could_capture_newline()`. When editing span code, note the four emission sites must stay consistent: the literal trie (`lit_table.rs`), `codegen/simple_regex.rs`, `codegen/full_regex.rs`, and the inline `illegal`/`parse_fn` blocks in `lib.rs`.
